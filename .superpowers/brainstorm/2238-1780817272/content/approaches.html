Architecture Approaches

Three ways to restructure the app for multi-job + pipeline view

A

重构 Layout + 新增 PipelineView（推荐）

保留现有路由页面代码，在 Layout 层面加入 Sidebar 和 StatusBar。新增一个 PipelineView 作为主页面，内嵌所有阶段的编辑面板。路由变为 / 是上传页、/pipeline/:jobId 是流水线视图。现有 CharacterEditor 等组件的逻辑抽取为可复用面板组件。

Pros

  • 改动可控，现有页面组件可复用
  • 渐进式迁移，不破坏已有功能
  • Sidebar/StatusBar 是全局组件，所有页面共享

Cons

  • 需要拆分为面板组件，有一定重构量
  • PipelineView 是全新页面，需从零开发
B

最小改动：每页面嵌入流水线面板

保留现有路由结构不变，在每个编辑页面的顶部加一个 PipelineProgress 组件，页面底部加 StatusBar。Sidebar 作为全局组件。不做 PipelineView 统一页面，用户仍通过路由跳转各阶段。

Pros

  • 改动最小，风险最低
  • 开发速度最快

Cons

  • 跟你的需求B（流水线内嵌编辑）矛盾
  • 没有统一的流水线视图体验
  • 断点重试的逻辑散落在各处
C

完全重建：单一 PipelineView + 状态机驱动

彻底重做前端，所有功能在一个 PipelineView 页面完成。引入状态机管理（XState 或 reducer），每个阶段为可展开面板，状态通过 context 全局共享。Sidebar 和 StatusBar 是视图的固定部分。

Pros

  • 架构最清晰，体验最统一
  • 状态管理强，断点重试天然支持

Cons

  • 工程量最大，需要重写大部分前端
  • 引入额外的状态管理库
  • 与现有代码几乎无复用
